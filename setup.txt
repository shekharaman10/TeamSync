Last updated: 2026-04-23


1. PROJECT OBJECTIVE
------------------------------------------------------------------------------

TeamSync is a multi-tenant B2B project management SaaS in the spirit of
Linear / Jira. Organizations create workspaces, invite members with role-based
access, organize work into projects and epics, and track tasks through a full
status lifecycle.

The system enforces clean architectural boundaries, database-level multi-tenant
isolation, and RBAC from day one — the kind of architecture that passes a
security review and scales to real customers.

STATUS: Fully implemented and running. Both backend (Express API) and frontend
(React SPA) are complete and integrated.


2. TECH STACK
------------------------------------------------------------------------------

Backend
  - Node.js 20+ + Express 5 (TypeScript strict mode)
  - Prisma 6 ORM + PostgreSQL 16
  - JWT access tokens (short-lived, httpOnly cookies)
  - Opaque refresh tokens (stored hashed in DB, rotated on use)
  - bcrypt (cost 10) for password hashing
  - zod for request validation and env validation
  - nodemailer (SMTP) for invitation and task notification emails
  - passport-google-oauth20 for Google OAuth

Frontend
  - Vite 5 + React 18 + TypeScript strict
  - TanStack Query v5 (server state, cache, pagination)
  - Zustand (client UI state — auth status, last workspace)
  - React Router v6 (file-based routing)
  - React Hook Form + zod resolver
  - Tailwind CSS v4
  - @dnd-kit/core + @dnd-kit/sortable (Kanban drag-and-drop)
  - recharts (analytics charts)
  - framer-motion (animations)

Database
  - PostgreSQL 16 (local dev on port 5432)

4. DATA MODEL
------------------------------------------------------------------------------

Source of truth: server/prisma/schema.prisma

Enums:
  Role:        OWNER | ADMIN | MEMBER
  TaskStatus:  BACKLOG | TODO | IN_PROGRESS | IN_REVIEW | DONE
  Priority:    LOW | MEDIUM | HIGH | URGENT
  AuthProvider: EMAIL | GOOGLE

Models (8):

  User          — id, email (unique), name, passwordHash, googleId,
                  provider, avatarUrl, createdAt, updatedAt

  Workspace     — id, name, slug (unique), ownerId → User, timestamps
                  CASCADE deletes all projects/tasks when deleted

  Membership    — userId → User, workspaceId → Workspace, role, joinedAt
                  @@unique([userId, workspaceId])
                  This table is the RBAC enforcement mechanism. Never
                  put role on the JWT. Look up per request.

  Project       — id, workspaceId → Workspace, name, key, description, timestamps
                  @@unique([workspaceId, key])  — key unique within workspace

  Epic          — id, projectId → Project (CASCADE), title, description, timestamps

  Task          — id, projectId → Project (CASCADE), epicId? → Epic (SET NULL),
                  title, description, status, priority, assigneeId?, createdById,
                  dueDate?, timestamps
                  @@index([projectId, status])
                  @@index([projectId, assigneeId])
                  @@index([projectId, priority])

  Comment       — id, taskId → Task (CASCADE), authorId → User, body, timestamps

  RefreshToken  — id, userId → User (CASCADE), tokenHash (SHA-256),
                  expiresAt, createdAt, revokedAt?

  Invitation    — id, email, workspaceId → Workspace (CASCADE), role,
                  token (unique, 32-byte random hex), invitedById → User,
                  expiresAt (7 days), acceptedAt? (null = pending), createdAt
                  @@index([email, workspaceId])


5. ENVIRONMENT VARIABLES (server/.env)
------------------------------------------------------------------------------

Required:
  DATABASE_URL              postgresql://USER:PASS@HOST:PORT/teamsync
  JWT_ACCESS_SECRET         64+ char random hex
  CLIENT_URL                http://localhost:5173 (dev)
  PORT                      5000

SMTP (email skipped silently if SMTP_HOST is blank):
  SMTP_HOST                 smtp.gmail.com
  SMTP_PORT                 587
  SMTP_SECURE               false
  SMTP_USER                 your@gmail.com
  SMTP_PASS                 Gmail app password (not your login password)
  SMTP_FROM                 TeamSync <your@gmail.com>

Optional (Google OAuth):
  GOOGLE_CLIENT_ID
  GOOGLE_CLIENT_SECRET
  GOOGLE_CALLBACK_URL

Other:
  COOKIE_DOMAIN             localhost (dev)
  NODE_ENV                  development

Generate a secret:
  node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"

Gmail app password: https://myaccount.google.com/apppasswords
  (Requires 2FA enabled on the Google account)


6. LOCAL SETUP (from scratch, new machine)
------------------------------------------------------------------------------

Prerequisites:
  - Node.js 20+
  - PostgreSQL 16 running on localhost:5432

Steps:

  # Clone and install
  git clone <repo>
  cd teamsync/server && npm install
  cd ../client && npm install

  # Create server/.env  (copy the variables from section 5 and fill them in)

  # Create the database
  psql -U postgres -c "CREATE DATABASE teamsync;"

  # Apply migrations and generate Prisma client
  cd server
  npx prisma migrate dev

  # Start both servers (two terminals)
  cd server && npm run dev        # API on :5000
  cd client && npm run dev        # SPA on :5173

  # Verify backend
  curl http://localhost:5000/api/health
  # → {"ok":true,"db":"connected","time":"..."}

  # Open the app
  http://localhost:5173

  # Sign up, create a workspace, then click "Load demo data" on the Projects
  # page to populate 2 projects, 6 epics, and 30 tasks.

9. ARCHITECTURAL PRINCIPLES
------------------------------------------------------------------------------

These rules must never be violated:

  1. RBAC lives in the database, not in tokens.
     Look up Membership per request. Role changes take effect immediately.

  2. Auth cookies are httpOnly, Secure (prod), SameSite=strict.
     No localStorage for auth tokens. No tokens in response bodies.

  3. Multi-table writes use Prisma transactions ($transaction).
     Create workspace → create owner Membership: atomic.
     Accept invitation → create Membership + mark accepted: atomic.

  4. Validate at the boundary with zod.
     Every controller parses req.body / req.params / req.query through
     a zod schema before touching the DB. Never trust raw input.

  5. Index filter columns.
     When adding a WHERE filter, confirm there's a @@index in schema.prisma.

  6. Cursor pagination, not offset.
     Use last-seen id + createdAt. Offset breaks silently at scale.

  7. Controllers are thin; services hold logic.
     Controllers: parse req → call service → respond.
     Services: Prisma calls + business rules. Testable without HTTP.

  8. Env validated at startup.
     Missing vars = immediate exit with a clear message. Never silent.

  9. No swallowed errors.
     Let errors propagate to the global handler. Log there.

  10. Multi-tenant isolation is a security requirement.
      Every workspace-scoped query filters by workspaceId from the
      authenticated Membership. Never trust workspaceId from the request body.
